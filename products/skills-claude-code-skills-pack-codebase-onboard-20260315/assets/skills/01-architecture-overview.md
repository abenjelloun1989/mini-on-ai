# Skill: Generate Architecture Overview

## Purpose

Produce a clear, accurate, and immediately useful architecture overview document
by deeply reading the actual codebase — not guessing or templating. Every claim
in the output must be grounded in files, configs, or code you have examined.

---

## Execution Steps

Follow these steps in order. Do not skip steps or combine them hastily.

### Step 1 — Orient to the Repository Root

- List the top-level directory contents
- Identify the repository type: monorepo, polyrepo, single service, library, etc.
- Note any workspace configuration files (`package.json` workspaces, `pnpm-workspace.yaml`,
  `Cargo.toml` workspace, `go.work`, `nx.json`, `turbo.json`, `lerna.json`, etc.)
- Record the primary programming languages present

### Step 2 — Identify the Technology Stack

Examine the following to build a verified stack list:

- Dependency manifests: `package.json`, `requirements.txt`, `pyproject.toml`,
  `Gemfile`, `go.mod`, `Cargo.toml`, `pom.xml`, `build.gradle`
- Runtime/infra config: `Dockerfile`, `docker-compose.yml`, `.tool-versions`,
  `.nvmrc`, `runtime.txt`
- CI/CD pipelines: `.github/workflows/`, `.circleci/`, `Jenkinsfile`, `buildkite.yml`
- Infrastructure as code: `terraform/`, `pulumi/`, `cdk/`, `k8s/`, `helm/`

Do NOT list a technology unless you found evidence of it in these files.

### Step 3 — Map System Components

For each significant directory or package:

- Read its `README.md` if present
- Identify its role: API server, worker, frontend app, shared library,
  database migration runner, CLI tool, etc.
- Note the entry point file (`main.go`, `index.ts`, `app.py`, `server.js`, etc.)
- Check for exposed ports or service definitions in compose/k8s files

If the repo is a monorepo, treat each workspace/package as a discrete component.

### Step 4 — Trace Data Flow

Look for evidence of how data moves through the system:

- HTTP/REST/GraphQL: route files, controller directories, OpenAPI specs
- Async/event-driven: queue configs, pub/sub topics, event bus setup,
  message broker env vars (Kafka, RabbitMQ, SQS, Redis Streams)
- Database interactions: ORM model directories, schema files, migration folders
- External integrations: third-party SDK imports, webhook handlers, API client files

Describe the primary request lifecycle in plain language (e.g., "A user request
hits the API gateway → routed to the auth service → validated → passed to the
core service → written to Postgres → response returned").

### Step 5 — Identify Service Boundaries

- List distinct deployable units (each Dockerfile or k8s Deployment = one unit)
- Note how services communicate: REST calls, gRPC, message queues, shared DB
- Flag any shared databases or caches accessed by multiple services
- Identify any API gateway, reverse proxy, or load balancer configuration

### Step 6 — Surface Key Design Decisions

Scan for architectural patterns and record what you find evidence of:

- Authentication/authorization approach (JWT, OAuth2, session cookies, API keys)
- Caching strategy (Redis, Memcached, CDN headers, in-memory)
- Background job processing (cron, workers, queues)
- Database choices and their apparent rationale if comments exist
- Feature flags or configuration management approach
- Observability setup (logging library, tracing, metrics exporters)

Only document decisions you found evidence for. If the rationale is unclear,
say so explicitly rather than inventing a reason.

### Step 7 — Write the Output Document

Produce a Markdown document structured exactly as shown in the Output Format
section below. Save it or present it for review — confirm with the user whether
to write it to a file.

---

## Output Format

The generated document must follow this structure:

```
# Architecture Overview: [Project Name]

> Generated by /gen-architecture on [date]. Review and update when major
> structural changes occur.

## 1. System at a Glance
One paragraph (4–8 sentences) describing what the system does, who uses it,
and the overall architectural style (e.g., monolith, microservices, serverless).

## 2. Technology Stack
| Layer | Technology | Notes |
|-------|-----------|-------|
List verified technologies only. Include version if found in config files.

## 3. Component Map
For each major component:
### [Component Name]
- **Type:** (API server / worker / frontend / library / etc.)
- **Location:** `path/to/component`
- **Responsibility:** One sentence.
- **Entry point:** `path/to/main/file`
- **Key dependencies:** List 2–4 most important internal or external deps

## 4. Data Flow
Narrative description + a simple ASCII or Mermaid diagram if the flow has
more than two hops. Label every arrow with the protocol or mechanism used.

## 5. Service Boundaries
Table or list of deployable units, their communication methods, and any
shared infrastructure (databases, caches, queues) they rely on.

## 6. Key Design Decisions
Bullet list. Each item: decision made, where evidence was found, and the
apparent rationale (or note that rationale is undocumented).

## 7. What's Not Covered Here
Be honest. List areas that are underdocumented in the code or that this
analysis could not fully resolve. Suggest follow-up questions for the team.
```

---

## Quality Rules

- **Ground every claim.** If you state that the system uses Redis, you must have
  seen Redis in a config, import, or environment variable. No assumptions.
- **Be concise.** Each component description should be readable in 30 seconds.
  Avoid prose that restates the directory name.
- **Use accurate paths.** All file/directory references must reflect the actual
  repo structure you explored.
- **Flag uncertainty clearly.** Use phrases like "appears to" or "based on X file"
  when you are inferring rather than observing directly.
- **No filler sections.** If a section has no findings (e.g., no background jobs
  exist), omit the section or write "None identified" — do not pad with generics.
- **Mermaid diagrams are optional but preferred** for systems with 3+ services.
  Only include if you can accurately label all nodes and edges.

## Usage Examples

### Example 1 — Generate overview for the entire repo

```
/gen-architecture
```

Claude will analyze the full repository from the root and write the architecture
overview to `docs/architecture-overview.md` (or prompt you for a save location).

---

### Example 2 — Scope to a single service in a monorepo

```
/gen-architecture services/payments
```

Claude will limit analysis to the `services/payments` directory, treating it as
the system boundary. Useful when onboarding someone who will work exclusively
on one service.

---

### Example 3 — Generate and immediately open for review

```
/gen-architecture --review
```

Claude will produce the document and display it inline in the terminal for
immediate review before writing to disk. Useful for a quick sanity check before
committing documentation.

---

## Notes for Claude

- If the repository has no README at all, note this prominently in Section 7
  as a documentation gap to address.
- If you encounter a monorepo with more than 8 packages, group packages by
  domain in Section 3 rather than listing each one individually.
- Always end your work by telling the user: (1) where the file was saved,
  (2) how long ago the oldest dependency manifest was updated (a freshness signal),
  and (3) one concrete suggestion for what documentation to create next.
