# Skill: Design Decisions Explainer

## Purpose

New engineers can read code to learn *what* it does. What they cannot easily
learn is *why* it does it that way — the constraints, failed experiments,
deliberate trade-offs, and business pressures that shaped the current
architecture. This skill surfaces that hidden context and packages it into
clear, durable documentation.

---

## How to Execute This Skill

### Step 1 — Parse the Invocation

Accept an optional scope argument from the user:

- `/explain-decisions` → analyze the entire repository
- `/explain-decisions src/payments` → focus on a specific directory or module
- `/explain-decisions --topic caching` → focus on a named topic or theme
- `/explain-decisions src/auth --depth deep` → deeper investigation with more
  git archaeology

Store the scope. Default to full-repo analysis if no argument is provided.

---

### Step 2 — Gather Evidence From Multiple Sources

Work through each evidence source below. Collect findings before writing output.

**2a. Git History Mining**

Run the following investigations:

- Identify the 20 most substantive commit messages (filter out "fix typo",
  "merge branch", formatting-only commits)
- Look for commits with messages containing keywords: `refactor`, `replace`,
  `migrate`, `switch`, `revert`, `decision`, `trade-off`, `because`, `instead`,
  `performance`, `security`, `simplify`, `workaround`, `hack`, `debt`
- Identify files with the highest churn rate — frequent rewrites signal
  contested or evolving decisions
- Find large single commits that replaced significant amounts of code
- Look for revert commits — these reveal decisions that were tried and undone

**2b. Inline Comment Mining**

Search source files for high-signal comment patterns:

- `// TODO`, `// FIXME`, `// HACK`, `// NOTE`, `// WARNING`, `// IMPORTANT`
- `// We use X instead of Y because`
- `// This is intentional`
- `# See issue #`, `# See PR #`, `// See ADR`
- Long block comments above classes, modules, or non-obvious functions
- Commented-out code blocks (signals something was tried and abandoned)

**2c. Configuration and Dependency Signals**

Examine:

- `package.json`, `go.mod`, `Cargo.toml`, `requirements.txt`, or equivalent —
  look for pinned versions, unusual dependencies, or notable absences
- Build configuration files for custom flags, excluded features, or workarounds
- `.env.example` or config schemas for non-obvious settings
- Feature flags or experiment configuration files

**2d. Architecture Pattern Recognition**

Scan code structure for patterns worth explaining:

- Unusual folder structures that deviate from framework conventions
- Custom implementations of things that libraries typically handle
- Abstraction layers that wrap external services or third-party code
- Error handling strategies that are applied consistently across the codebase
- Data flow patterns (e.g., event-driven vs. direct calls in unexpected places)

---

### Step 3 — Cluster Findings Into Decisions

Group your evidence into discrete design decisions. Each decision should
represent one coherent choice the team made. Aim for 5–12 decisions for a
focused scope, 10–20 for a full repository.

For each candidate decision, assess:

- **Confidence** (High / Medium / Low) — how well-evidenced is this?
- **Impact** (High / Medium / Low) — how much does this affect daily development?
- **Age** — is this decision recent or potentially outdated?

Prioritize High Impact decisions regardless of age. Flag Low Confidence
findings explicitly.

---

### Step 4 — Write the Design Decisions Document

Produce a Markdown document using the structure below.

---

**Output Structure:**

# Design Decisions: [Scope or Repository Name]

> Generated by `/explain-decisions` on [date].  
> Sources: git history ([N] commits analyzed), inline comments, dependency files,
> code patterns.  
> Scope: [full repo / specific path / topic]

---

## How to Read This Document

Brief 2–3 sentence note explaining that these are *reconstructed* decisions
based on evidence — not official records. Where confidence is low, the
document says so. Engineers should treat this as a starting point for
conversations, not a source of truth.

---

## Decision Index

A numbered list of all decisions covered, so readers can jump to what matters
most.

---

## Decision [N]: [Short Descriptive Title]

**Impact:** High / Medium / Low  
**Confidence:** High / Medium / Low  
**Area:** [e.g., Data Layer, API Design, Authentication, Build System]  
**Evidence Sources:** [e.g., commit abc1234, comments in src/db/pool.ts, dependency pinning]

### What Was Decided

One clear paragraph stating the decision in plain terms. What is the current
approach? What does the code actually do?

### Context and Constraints

What pressures, constraints, or circumstances led to this decision? Include:
- Timeline pressures if evident from commit density
- External constraints (library limitations, API restrictions, compliance needs)
- Team size or expertise signals
- Prior state if a migration is evident

### Alternatives That Were Considered

What else was tried or considered? This section may be speculative when
evidence is limited — label it accordingly.

### Trade-offs

| Benefit | Cost |
|---------|------|
| [what this approach gains] | [what this approach sacrifices] |

### What to Watch For

One or two sentences on what future engineers should know before changing this.
Are there hidden dependencies? Downstream effects? Known limitations to be
aware of?

---

*Repeat the Decision block for each finding.*

---

## Potentially Outdated Decisions

List any decisions that show signs of age — the original constraint may no
longer apply, the library has been superseded, or the team has grown past the
original assumption. Flag these for review, not immediate action.

## Low-Confidence Observations

List patterns noticed that *might* be intentional decisions but lack clear
evidence. Frame these as questions: "It appears that X is the case — is this
intentional?"

## Recommended Follow-Up

List 3–5 specific questions engineering leads should discuss with the team
to validate or update the findings in this document.

---

### Step 5 — Validate Before Delivering

Before presenting output, check:

- [ ] Every High Impact decision has at least one concrete evidence source cited
- [ ] Speculation is clearly labeled (phrases like "this appears to be" or
      "based on commit patterns")
- [ ] No decision is stated as certain when confidence is Medium or Low
- [ ] Technical jargon is briefly explained the first time it appears
- [ ] The document would make sense to a competent engineer who is new to this
      specific codebase

---

## Output Format Rules

- Output a single Markdown document, ready to save as `DECISIONS.md`
- Use the exact section structure defined in Step 4
- Keep each decision write-up between 150–350 words
- Use plain language — avoid hedging so much that nothing useful is communicated
- Cite specific evidence (file paths, commit hashes, line references) wherever
  possible
- Do not fabricate decisions — if evidence is thin, say so explicitly

---

## Constraints

- Do not claim certainty about intent you cannot verify from the codebase
- Do not expose secrets, credentials, or PII found in history or comments
- If a decision appears to be a security-sensitive choice, note it clearly but
  do not publish the specific vulnerability details inline
- Keep the total document under 3,000 words for focused scopes; under 5,000
  words for full-repo analysis

---

## Usage Examples

**Example 1 — Full repository analysis**

/explain-decisions

Analyzes the entire repository and produces a full DECISIONS.md covering the
most impactful architectural choices found across git history, comments, and
code patterns.

---

**Example 2 — Focused module analysis**

/explain-decisions src/billing

Scopes all investigation to the billing module. Useful when onboarding an
engineer who will own a specific domain, or before a targeted refactor of
that area.

---

**Example 3 — Topic-focused deep dive**

/explain-decisions --topic database --depth deep

Investigates all database-related decisions across the codebase with deeper
git archaeology. Surfaces decisions about ORM choice, connection pooling
strategy, migration tooling, query patterns, and data modeling trade-offs.
