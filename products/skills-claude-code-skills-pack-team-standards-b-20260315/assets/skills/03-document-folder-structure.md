# Skill: Document Folder Structure (`/doc-structure`)

## Purpose

Produce a richly annotated folder structure map that explains the **rationale**
behind each directory — its purpose, its naming convention, and the
organizational principle it reflects. The output helps new hires build a mental
model of the codebase quickly, without needing to reverse-engineer intent from
file contents alone.

---

## Execution Instructions

Follow these steps in order every time `/doc-structure` is invoked.

### Step 1 — Discover the Directory Tree

1. Run `find . -type d` (or equivalent) from the repository root, excluding
   common noise directories: `.git`, `node_modules`, `.next`, `dist`, `build`,
   `__pycache__`, `.venv`, `coverage`, `.cache`.
2. Capture the full tree up to **4 levels deep** by default. If the user
   specifies a path argument (e.g., `/doc-structure src/`), scope discovery
   to that subtree.
3. Note directories that contain a disproportionately large or small number
   of files — these often signal architectural intent worth calling out.

### Step 2 — Gather Supporting Context

For each discovered directory, collect the following signals to infer rationale:

- **README or index files** — read any `README.md`, `index.ts`, `__init__.py`,
  or `mod.rs` at the directory root for explicit documentation.
- **File naming patterns** — look at filenames inside the directory to infer
  the category of content (e.g., `*.test.ts`, `*.controller.ts`, `schema_*.sql`).
- **Import/export patterns** — check whether files in this directory are
  imported by other directories, suggesting a shared/utility role.
- **Config files** — detect tooling config files (`jest.config.js`,
  `Dockerfile`, `.eslintrc`) that explain a directory's operational role.
- **Git history signals** — if available, note if a directory changes
  frequently (volatile) or rarely (stable/foundational).

### Step 3 — Infer and Draft Rationale

For each directory, write a rationale annotation covering three dimensions:

1. **Purpose** — What kind of code or assets live here? What problem does
   this directory solve?
2. **Naming choice** — Why is it called what it's called? Is it a framework
   convention, a domain term, or a team-specific shorthand?
3. **Organizational principle** — What rule governs what belongs here vs.
   somewhere else? (e.g., "co-location by feature", "separation of
   concerns", "layer-based architecture")

If rationale cannot be confidently inferred, mark it with `[Inferred — verify
with team]` so new hires know to ask.

### Step 4 — Produce the Annotated Map

Render the output using the **exact format** specified below.

---

## Output Format

The output is a single Markdown document suitable for saving as
`FOLDER_STRUCTURE.md` at the repository root (or the scoped subtree root).

Structure the document as follows:

---

FOLDER_STRUCTURE.md

# Folder Structure

> Generated by `/doc-structure` — annotates directory purpose, naming rationale,
> and organizational principles to help new team members build a mental model
> of the codebase.

## Overview

[2–4 sentence summary of the top-level architectural philosophy — e.g.,
feature-based, layer-based, monorepo with workspaces, etc.]

## Directory Map

Use indented tree notation followed by an annotation block for every
significant directory. Format each entry as:

📁 path/to/directory/
Purpose: One sentence describing what lives here.
Naming: Why this name was chosen.
Principle: The organizational rule governing membership in this directory.
Note: [Optional — edge cases, historical context, or [Inferred — verify with team]]

Recurse through all levels discovered in Step 1. Skip generated or
transient directories (dist, build, __pycache__) — list them once in
a "Generated Directories" section at the bottom with a single-line note.

## Organizational Principles Glossary

List the distinct principles referenced in the map (e.g., "Feature
co-location", "Separation of concerns") with a 1–2 sentence definition each,
so new hires understand the vocabulary.

## Questions to Ask Your Team

Auto-generate 3–5 questions targeting directories flagged as
[Inferred — verify with team] or directories that show unusual patterns.

---

## Constraints and Quality Rules

- **No invented rationale.** If you cannot infer purpose from available
  signals, say so explicitly with `[Inferred — verify with team]`.
- **Annotate every non-trivial directory.** Do not silently skip directories
  unless they are generated/transient artifacts.
- **Keep annotations concise.** Purpose, Naming, and Principle entries must
  each be a single sentence. Use the Note field for anything longer.
- **Preserve actual directory names exactly.** Do not rename or normalize
  directory names in the output.
- **Flag anti-patterns.** If a directory structure appears to violate its
  own stated principle (e.g., a "utils" folder containing domain logic),
  add a Note flagging the inconsistency — do not silently ignore it.
- **Scope matters.** When a path argument is provided, restrict output to
  that subtree and adjust the Overview summary accordingly.
- **Do not modify any files** unless the user explicitly confirms they want
  the output written to `FOLDER_STRUCTURE.md`.

---

## Usage Examples

### Example 1 — Full repository scan

/doc-structure

Analyzes the entire repository from the current working directory and
produces a complete annotated map of all directories up to 4 levels deep.

### Example 2 — Scoped to a subdirectory

/doc-structure src/

Restricts analysis to the `src/` subtree only. Useful when the repository
is large and you want onboarding docs focused on application code rather
than infrastructure or tooling directories.

### Example 3 — Save output to file

/doc-structure --save

Runs the full scan and, after displaying a preview, writes the annotated
map directly to `FOLDER_STRUCTURE.md` at the repository root. Prompts
for confirmation before writing.

## Notes for Maintainers

- Re-run this skill after any significant directory restructuring.
- Pair the output with `/contributing` (generates `CONTRIBUTING.md`) and
  `/decision-log` (captures architectural decision rationale) for a complete
  new-hire onboarding documentation suite.
- If your team uses a monorepo tool (Turborepo, Nx, Lerna), signal this in
  the initial prompt — the skill will adjust its Overview summary and
  highlight workspace boundaries explicitly.
