# Skill: infer-coding-conventions

## Purpose

Extract the *actual* coding conventions in use — not what anyone said they do,
but what the code itself demonstrates. Output a structured, human-readable
summary that an engineering lead can review, edit, and hand directly to new
team members.

---

## When to Use

- Before generating a CONTRIBUTING.md (pairs with `/generate-contributing`)
- When onboarding new engineers who need to match existing style
- When auditing whether stated conventions match real usage
- After acquiring or inheriting a codebase with no documentation

---

## Execution Steps

Claude must follow these steps in order. Do not skip steps or combine them
into a single pass.

### Step 1 — Establish Scope

1. Identify the primary language(s) in the repo by scanning file extensions.
2. List the top-level directory structure.
3. Note any existing convention documents (`.editorconfig`, `eslint.config.*`,
   `.prettierrc`, `pyproject.toml`, `rubocop.yml`, `CONTRIBUTING.md`, etc.).
   Read them but treat them as *stated* conventions — verify them against code.
4. Select a representative sample of files to analyze:
   - At least 10 source files spanning different modules or packages
   - Prefer files with >50 lines
   - Include at least one test file and one config file if present

### Step 2 — Analyze Naming Patterns

Examine the sample files and record observed patterns for:

- **Variables**: camelCase, snake_case, SCREAMING_SNAKE, Hungarian, etc.
- **Functions / Methods**: naming style, verb-first vs noun-first, boolean
  prefixes (`is`, `has`, `should`, `can`)
- **Classes / Types / Interfaces**: PascalCase, prefix conventions (`I`, `T`,
  `Abstract`), suffix conventions (`Service`, `Controller`, `Manager`, `Util`)
- **Files and folders**: naming style, index file usage, barrel exports
- **Constants and enums**: casing, grouping conventions
- **Test files**: suffix (`*.test.*`, `*.spec.*`, `_test.*`), co-location vs
  separate `__tests__` directory

For each pattern, note the **observed frequency** (dominant / mixed / rare).

### Step 3 — Analyze Code Structure Patterns

Examine how code is organized and note:

- **Import ordering**: stdlib → third-party → internal? Alphabetical within
  groups? Enforced by tooling or convention?
- **Export style**: named vs default exports, barrel files, re-export patterns
- **Function length and decomposition norms**: are functions kept short, or are
  longer procedural blocks common?
- **Error handling**: exceptions vs result types vs callbacks vs `null` returns
- **Async patterns**: callbacks, Promises, async/await, observable streams
- **Comment style**: inline comments, JSDoc/docstrings, TODO format, ratio of
  comments to code
- **Magic numbers/strings**: extracted to constants or inline?
- **Dependency injection vs direct imports**: pattern used for services/modules

### Step 4 — Analyze Folder and Module Structure

Map the project layout and infer:

- What lives at the top level vs nested?
- Is structure **feature-based**, **layer-based** (controllers/services/models),
  or **type-based** (components/hooks/utils)?
- Are shared utilities centralized (`lib/`, `utils/`, `shared/`) or
  co-located?
- Where do tests live relative to source?
- Where do types/interfaces live?
- How are environment-specific files handled?

### Step 5 — Cross-Reference Tooling

Check tooling configs to flag where conventions are **enforced** vs
**conventional only**:

- List active linters and formatters with their config files
- Note any rules that deviate from defaults (these signal intentional choices)
- Flag conventions the tooling does NOT cover (pure team convention)

### Step 6 — Identify Inconsistencies

Before writing the summary, note any areas where the codebase is **not**
consistent. These are important for the output — they tell leads where
documentation or enforcement is needed most.

### Step 7 — Produce Structured Output

Write the conventions summary in the format defined below.

---

## Constraints and Quality Rules

- **Ground every claim in observed code.** Do not invent conventions or
  default to language community norms unless explicitly noting you are doing so.
- **Quote real examples** for each named convention — at least one concrete
  code snippet or identifier.
- **Never state a convention as absolute** if you found counter-examples —
  use "predominant" or flag with ⚠️.
- **Keep the output editable.** Engineering leads will modify this. Use plain
  Markdown, no complex formatting that breaks in editors.
- **Be concise in descriptions.** Each pattern description should be 1–3 lines.
  Save depth for the inconsistencies section.
- If the codebase is very large (>500 source files), sample systematically:
  pick files from each major module rather than only top-level files.

---

## Usage Examples

### Basic — infer conventions for the whole repo

/infer-conventions

Claude scans the entire codebase and produces a full conventions summary
saved to `docs/inferred-conventions.md`.

---

### Scoped — infer conventions for one module or service

/infer-conventions src/payments

Claude limits analysis to the `src/payments` directory, useful for
monorepos where different services may have diverged in style.

---

### Comparison — check if stated conventions match actual code

/infer-conventions --compare CONTRIBUTING.md

Claude reads the existing `CONTRIBUTING.md` as the stated standard, then
compares observed patterns against it and produces a gap report: which
stated conventions are followed, which are ignored, and which real
conventions are undocumented.

---

## Output Destination

By default, write output to `docs/inferred-conventions.md`. If `docs/` does
not exist, write to the project root as `INFERRED-CONVENTIONS.md` and note
the location in the final response.
